# X68000用ゲームソフト『BLOCK BREAK PRO-68KZ』(ver. 2.00)

## 概要

BOOTHで公開中のX68000用ゲームソフト [BLOCK BREAK PRO-68KZ](https://sendac.booth.pm/items/6447331) のソースコード一式です。

内容に関しては、ブロック崩しゲームとなります。  
詳しくは、同梱テキストの [manual.txt](manual.txt) をご覧ください。

## 開発の経緯

昨今の多くのプログラミング環境では、コンピュータの動作原理が抽象化されブラックボックスとして扱われています。

これに対し、1980年代に発売されたパーソナルコンピュータ「X68000」は、プログラマが回路を読み解き、CPUおよび周辺LSIの直接制御を行う文化が先人たちにより形成されていました。

2023年、瑞起によりX68000の復刻版である [X68000 Z](https://www.zuiki.co.jp/x68000z/) が発売されたこと、および、同製品のコミュニティサイト Z-CLUB が開設され、先人たちの知見が現代において再び共有されるようになったことを契機として、この技術を継承したいという思いから本作の開発を開始しました。

## 開発の意図

X68000でのプログラミングといえばアセンブリ言語による開発のイメージが強いですが、本作ではC言語でどこまで実用的かつ快適なゲーム体験を実現できるのかを検証することを主眼として開発を行いました。

また、開発開始以前に周辺LSIの直接制御に関する個別検証を行ってきましたが、これらドライバ層をアプリケーションを下支えするライブラリ群として、どのように実装・統合するのかという設計面も大きなテーマでした。

さらに、意図的にボールやアイテムが大量に出現するゲームデザインとすることで、処理負荷の変化を観察しながら、パフォーマンスチューニング技術の習得を目指しました。

## 検証結果

本作のリリース後、Z-CLUB コミュニティサイト上では1年以上にわたりスコアアタックが継続して行われ、ランキングの更新が続きました。

また、BOOTHでの配布においては120ダウンロードを超え、多くの方々から好意的な評価をいただくことができました。

この結果から、C言語による実装であっても、適切な設計を行うことで十分実用的なゲーム体験を提供できることが確認できました。

開発当初から重視していた操作感やプレイフィールについて直接言及されることは多くありませんでしたが、長期間にわたり熱心に遊んでいただけたこと自体が、ここまでの試みの成果を示す裏付けであると考えています。

プレイしてくださった皆様に、この場を借りて感謝申し上げます。

## 開発における技術的制約と工夫

本作の開発は、以下のようにハードウェアの制約がある中で、いかにしてゲーム仕様を実現するかという部分に多くの時間を割いています。

### ブロックの表示方法について

Windows向けの開発では、背景・スプライト・UI要素など、全ての描画を統一されたパイプライン上で扱うことが可能であり、潤沢なリソースを用いてオブジェクト構造を抽象化した一元管理を行う設計が成り立ちます。

一方、X68000向けの開発においては、以下のように描画系がハードウェアレベルで分離されています。

- テキスト画面
- グラフィック画面
- スプライト
- BG面 (タイル)

本作は、ボールとアイテムが大量に出現するゲームデザインとなっていますが、スプライトは128枚までしか同時に表示することができません。

このため、ブロックに関してはスプライトではなくBG面を用いた描画を行う方針としましたが、BG面は16×16ドット固定タイルであるため、ブロック(32×8ドット)をそのまま表現することができず、以下の構造で対応しています。

- 横方向：2タイルを1ブロックとして扱う
- 縦方向：1タイル内に上下2ブロックを配置 (ブロック8種の組み合わせ64パターンの画像を選択して表示)

### グラフィック画面の描画速度について

Windows向けの開発では、大きな画像を一瞬で表示することが可能ですが、X68000ではテキスト画面またはグラフィック画面の表示範囲全体に描画を行った場合、描画途中の状態が可視化されてしまいます。

本作には、タイトル・プレイ・リザルトと3つのシーンが存在し、このシーンが遷移する際にグラフィック画面の背景画像を切り替えていますが、非常に見栄えの悪いものとなってしまいました。

このため、VRAMへの書込みにDMA転送を利用したところ劇的に速度が向上したものの、まだ一瞬の描画途中状態が見えてしまうため、以下の対策を行いました。

- シーン遷移時にグラフィックパレットを全て黒のものに切り替える
- 描画完了後に本来のパレットへ復帰する

また、画像ファイルのロードもWindows向けの開発ではシーン遷移時に行っていましたが、本作ではアプリケーション起動時に一括ロードを行い、事前に全てグラフィック画面へ描画しておき、シーン遷移時は表示ページを切り替えるという手法を取りました。

これにより、起動時に若干の待ち時間を要するものの、以降は待ち時間のない非常にスムーズなプレイが可能となりました。

## フォルダ(ディレクトリ)構成

```text
.
│
├─ image         画像データ
│   ├─ back      背景画像
│   ├─ block     ブロック画像
│   ├─ item      アイテム画像
│   ├─ num       数字画像
│   ├─ racket    ラケット画像
│   └─ single    その他単体画像 (ボール, カーソル, ショット)
│
├─ src           ソースコード
│   ├─ config    設定関連 (画像ファイル名, スプライト割り当て, PCM効果音番号)
│   ├─ lib       ドライバ層
│   │   ├─ crt   ディスプレイ制御
│   │   ├─ dma   DMA転送制御
│   │   ├─ draw  描画制御 (グラフィック画面, テキスト画面, スプライト)
│   │   ├─ input キーボード及びジョイスティック制御
│   │   └─ timer 時間制御 (IOCS ONTIME()使用)
│   ├─ object    アプリ層：ゲームオブジェクト (ボール, ブロック, ショット, アイテム など)
│   └─ scene     アプリ層：各シーン (タイトル, プレイ, リザルト)
│
└─ zpd           Z-MUSICで使用するZPDファイルと基となったPCMデータ
```


## ビルド方法

本作は以下の開発環境・ツールチェーンを使用してビルドを行っています。

- Human68k
- GCC真里子版 (ver1.42)
- XCライブラリ
- Z-MUSIC (v2.00)

`mklib.bat` を実行すると、ドライバ層（libディレクトリ）のみがコンパイルされ `GAMELIB.L` が生成されます。

`GAMELIB.L` がルートディレクトリにある状態で `make` を実行すると、アプリ層がコンパイルされ `BB68.x` が生成されます。

`rebuild.bat` を実行すると、ドライバ層とアプリ層を一括でコンパイルし `BB68.x` が生成されます。

## ビルド生成物の起動方法

BB68.x の起動をするには [src/config/image_filename.h](src/config/image_filename.h) にて設定しているパスに使用する画像ファイルが全て存在している必要があります。

また、起動前に Z-MUSIC を常駐させ BBPRO68K.ZPD を読み込んでおく必要があります。

[BOOTH版](https://sendac.booth.pm/items/6447331) は、すぐに起動可能なフロッピーディスクイメージ(.xdf)形式になっていますので、このイメージ内の BB68.x を差し替えれば簡単に起動することが可能です。
